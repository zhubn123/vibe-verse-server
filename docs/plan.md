# AetherFlow WMS Plan

## 协作方式

- 大方向只看阶段：阶段一 / 阶段二 / 阶段三。
- 日常开发由“代办池”驱动，不在阶段描述里展开实现细节。
- 代办状态统一标记：`[TODO]`、`[DOING]`、`[DONE]`。
- 规则：代办不可删除；完成后改状态并补一行“完成说明”。

---

## 阶段总览

### 阶段一（MVP 闭环）

- 建立 WMS 核心业务闭环：主数据 -> 入库 -> 出库 -> 库存查询。
- 完成最小可用权限能力（登录 + 基础鉴权）。
- 核心目标：业务跑通、数据一致、可稳定演示。

### 阶段二（可配置与可复制）

- 补齐字典、参数、流程配置能力（减少硬编码）。
- 强化主数据联动、规则校验、异常处理。
- 核心目标：不同场景通过配置复用，不靠代码分叉。

### 阶段三（可视化与扩展）

- 引入空间模型与可视化能力（含 2D/3D 展示准备）。
- 评估与 ERP/WCS/外部系统协同能力。
- 核心目标：从“能用”进化到“可扩展、可产品化”。

### [DONE] TODO-A008 出库库存校验与并发扣减

- 背景：
  出库是 WMS 中最容易出问题的环节，必须防止库存不足、重复提交、并发超扣。
- 目标：
  出库扣减库存安全可靠。
- 待办：
  1. 出库前校验可用库存是否足够。
  2. 扣减库存使用条件更新：`where stock_qty >= out_qty`。
  3. 防止重复提交同一出库单。
  4. 出库失败时不写流水、不改变库存。
  5. 增加异常提示：库存不足、库位无库存、物料不匹配。
- 验收标准：
  1. 库存不足时出库失败。
  2. 并发出库不会出现负库存。
  3. 重复点击提交不会重复扣减。
- 完成说明：
  - 已将出库确认改为“单据级防重 + 库存原子扣减”组合：同一出库单确认时先加短锁，再执行数据库条件更新扣减库存。

  - 已补精确异常提示：区分 `库位无库存`、`库位物料不匹配`、`库存不足`、`重复确认处理中`。

  4. 前端用户能直观看到状态并理解当前可执行操作。
- 完成说明：
  - 已统一收敛为 `draft`、`confirmed` 两态，创建默认草稿，确认后才真正影响库存。

### [DONE] TODO-A010 库存调整 MVP（先不做盘点单）

- 背景：
  真实仓储里库存不可能永远准确，阶段一需要一条“受控纠偏”通道；但完整盘点流程涉及冻结、账面数/实盘数对比、差异生成与复盘，不适合和库存调整一起一次性做重。
- 目标：
  先补齐库存调整单最小闭环，满足人工增减库存、可追溯、与库存流水一致。
- 口径收敛（阶段一先固定）：
  1. `A010` 阶段一只做“库存调整单”，暂不做独立盘点单。
  2. 库存调整同样沿用现有两态：`draft`、`confirmed`。
  3. 只有 `confirmed` 状态才真正影响库存。
  4. 盘点单、库存冻结、账实差异自动生成调整、批量盘点导入暂不纳入本任务。
- 待办：
  1. 新增库存调整单与明细，最少字段包含：仓库、区域/库位、物料、调整方向、调整数量、调整原因、备注。
  2. 支持两种调整方向：增加库存、减少库存；减少库存时必须复用现有可用库存校验，不能出负库存。
  3. 创建默认草稿；确认后统一走 `InventoryService.applyStockChanges(...)` 更新库存并写库存流水。
  4. 调整原因必填，原因描述以业务字段留痕，不只写到 remark。
  5. 已确认调整单不允许编辑、删除；草稿单允许编辑、删除。
  6. 前端列表显示状态与调整方向，按状态禁用无效操作按钮。
  7. 为后续盘点能力预留业务类型和扩展点，但不在本任务内落完整盘点流程。
- 验收标准：
  1. 管理员可通过库存调整单对指定仓库/库位/物料做受控增减。
  2. 减少库存时会校验可用库存，不允许调整成负数。
  3. 所有调整都能在库存流水中追溯到来源单据、操作人、调整原因和前后数量。
  4. 已确认调整单前后端都不能再编辑或删除。
- 完成说明：
  - 已新增 `inventory_adjustment`、`inventory_adjustment_item` Liquibase 变更，以及库存调整单后端实体、Mapper、Service、Controller 全链路。
  - 调整方向已收敛为 `INCREASE`、`DECREASE`；明细 `adjust_qty` 始终保存正数，确认时由 `adjust_type` 决定库存正负变化。
  - 库存调整确认已统一走 `InventoryService.applyStockChanges(...)`，减少库存时复用现有可用库存校验与条件扣减逻辑，库存流水可追溯来源单据与调整原因。
  - 已确认库存调整单前后端都禁止编辑、删除；前端已补库存调整页面、路由、导航入口与状态/方向展示。
  - 已补 `InventoryAdjustmentA010IntegrationTests`，覆盖增加库存成功、减少库存不足拦截、已确认单据不可删除。

  4. `GET /users/profile`：获取个人信息。
  5. `PUT /users/profile`：更新个人信息。
  6. `PUT /users/password`：修改密码（旧密码校验）。
- 安全与规则：
  1. 密码存储使用 `BCrypt` 哈希，禁止明文与可逆加密。
  2. 登录失败次数限制（如 5 次）与短时锁定策略。
  3. 注册默认角色（如 `operator`）可配置；`admin` 仅人工分配。
  4. 登录态仅保存必要信息（`userId` 等），权限按角色解析。
- 非目标（本阶段明确不做）：
  1. 不做组织架构（部门/岗位）与数据权限细粒度模型。
  2. 不做第三方登录（OAuth/SSO）与多因子认证（MFA）。
  3. 不做复杂审批流账号开通（先保证直连闭环）。
  4. 不做多租户账号体系（后续如需产品化再单独规划）。
- 完成定义（DoD）：
  1. 前端注册页、登录页、个人信息页全链路可用，去除临时占位分支。
  2. 鉴权在网关/拦截器层统一生效，业务接口不出现“漏拦截”。
  3. 登录、改密、资料修改具备审计记录（含操作者、时间、结果）。
  4. 关键异常口径统一（账号不存在、密码错误、账号锁定、无权限）。
  5. 形成文档化接口契约（请求/响应/错误码），作为后续冻结基线。
