# WMS 项目开发计划（第一批优先版）

## 1. 项目定位

项目名称：

- 智能仓储与资产流转管理系统（AetherFlow WMS）

项目目标：

- 先做一版自己的轻量 WMS，不走开源项目二改路线。
- 第一批先跑通仓储主流程闭环，登录权限放在第一批收尾阶段统一接入。
- 形成可演示、可迭代、可写进简历和毕设说明的基础版本。

当前技术方向：

- 后端：Spring Boot 3 + MyBatis-Plus + MySQL
- 前端：Vue 3 + Vite
- 文档：Swagger / OpenAPI
- 环境：Docker Compose

## 2. 本轮策略（按你当前要求）

第一批开发顺序固定为：

1. 仓库管理
2. 库位管理
3. 物料管理
4. 入库管理
5. 出库管理
6. 库存查询
7. 登录与权限（第一批结束做）

说明：

- 先把业务数据流和库存逻辑打通，再加鉴权层。
- 第一批业务阶段默认内部调试环境可直连接口。
- 收尾时统一接入登录、角色、菜单、接口权限控制。

## 3. 业务关系与流程口径（整体到局部）

### 3.1 模块关系

1. 仓库管理：定义仓储业务容器。
2. 库位管理：定义仓库内部位置，必须隶属某个仓库。
3. 物料管理：定义被管理的货物种类。
4. 入库管理：记录物料进入仓库/库位的业务单据。
5. 出库管理：记录物料离开仓库/库位的业务单据。
6. 库存查询：展示库存结果，口径来自入出库累计。

### 3.2 核心业务关系

- 一个仓库可包含多个库位。
- 一个库位只属于一个仓库。
- 一个物料可分布在多个仓库和多个库位。
- 入库与出库均采用“单头 + 明细”业务结构。
- 库存唯一口径采用：仓库 + 库位 + 物料。

### 3.3 第一批业务流程

1. 建立主数据：仓库、库位、物料。
2. 创建并确认入库单，库存增加。
3. 创建并确认出库单，库存校验通过后扣减。
4. 在库存查询中按仓库、库位、物料查看结果。

### 3.4 第一批关键约束

- 出库前必须校验库存，库存不足不允许出库。
- 仓库停用前需检查是否有库存或在途单据。
- 库位停用前需检查是否有库存。
- 物料停用后不允许新增入出库明细。

## 4. 第一批需求分析

### 3.1 核心业务目标

第一批必须跑通的闭环：

1. 建仓库
2. 建库位并绑定仓库
3. 建物料
4. 创建入库单并完成入库
5. 创建出库单并完成出库
6. 查询库存结果

### 3.2 角色范围（第一批）

第一批业务阶段先只考虑“操作员”单角色行为，不做复杂权限分流。

在第一批收尾再补：

- 登录
- 角色
- 菜单
- 接口权限

### 3.3 关键业务规则

- 入库完成后，库存增加。
- 出库前必须校验可用库存是否足够。
- 出库完成后，库存减少。
- 库位必须属于某个仓库。
- 入库单与出库单都必须绑定物料。
- 第一批暂不处理批次、效期、冻结库存、多计量单位。

## 5. 第一批功能清单（完善补充）

### 4.1 仓库管理

最小能力：

- 新增仓库
- 编辑仓库
- 查询仓库列表
- 仓库启停状态

### 4.2 库位管理

最小能力：

- 新增库位（必须选择仓库）
- 编辑库位
- 查询库位列表
- 按仓库过滤库位

### 4.3 物料管理

最小能力：

- 新增物料
- 编辑物料
- 查询物料列表
- 物料启停状态

第一批定义：

- 物料先作为“货物种类登记”，不做复杂 SKU 属性模型。

### 4.4 入库管理

最小能力：

- 新建入库单
- 入库单明细（物料、数量）
- 入库确认（影响库存）
- 入库单列表查询

### 4.5 出库管理

最小能力：

- 新建出库单
- 出库单明细（物料、数量）
- 出库确认（扣减库存）
- 出库单列表查询

### 4.6 库存查询

最小能力：

- 按仓库查询库存
- 按物料查询库存
- 展示当前库存数量

### 4.7 登录与权限（第一批收尾）

最小能力：

- 用户登录
- 用户登出
- 角色基础模型（admin/operator）
- 接口基础鉴权

## 6. 第一批数据模型范围

建议第一批最少表：

- `warehouse`（仓库）
- `location`（库位）
- `material`（物料）
- `inbound_order`（入库单）
- `inbound_order_item`（入库单明细）
- `outbound_order`（出库单）
- `outbound_order_item`（出库单明细）
- `inventory`（库存）

第一批收尾补：

- `sys_user`
- `sys_role`
- `sys_user_role`
- `sys_menu`（可简化）
- `sys_role_menu`（可简化）

## 7. 第一批接口规划（建议）

仓库：

- `GET /api/warehouses`
- `POST /api/warehouses`
- `PUT /api/warehouses/{id}`

库位：

- `GET /api/locations`
- `POST /api/locations`
- `PUT /api/locations/{id}`

物料：

- `GET /api/materials`
- `POST /api/materials`
- `PUT /api/materials/{id}`

入库：

- `GET /api/inbound-orders`
- `POST /api/inbound-orders`
- `POST /api/inbound-orders/{id}/confirm`

出库：

- `GET /api/outbound-orders`
- `POST /api/outbound-orders`
- `POST /api/outbound-orders/{id}/confirm`

库存：

- `GET /api/stocks`

收尾鉴权：

- `POST /api/auth/login`
- `POST /api/auth/logout`

## 8. 第一批实施顺序（详细）

### 7.1 后端顺序

1. `modules/wms/warehouse`
2. `modules/wms/location`
3. `modules/wms/material`
4. `modules/wms/inbound`
5. `modules/wms/outbound`
6. `modules/wms/inventory`
7. `modules/system/auth`（收尾接入）

### 7.2 前端顺序

1. 仓库页面
2. 库位页面
3. 物料页面
4. 入库页面
5. 出库页面
6. 库存页面
7. 登录页与权限路由守卫（收尾）

### 7.3 联调顺序

1. 先通仓库、库位、物料基础 CRUD
2. 再通入库和库存联动
3. 再通出库和库存校验
4. 最后统一加登录和接口鉴权

## 9. 验收标准（第一批）

第一批完成必须满足：

- 能新增仓库、库位、物料
- 能完成入库并增加库存
- 能完成出库并扣减库存
- 出库库存不足时有明确错误提示
- 能按仓库和物料查询库存
- 最后可登录后访问业务接口（未登录拒绝）
- Swagger 可展示核心接口

## 10. 第一批暂不做

- 微服务拆分
- 审批流
- 报表中心
- 盘点与调拨
- Redis 缓存优化
- 多租户
- 复杂数据权限

## 11. 风险与控制

主要风险：

- 先无权限开发，后补权限时接口改动过大
- 入/出库与库存更新逻辑不一致
- 数据模型过早复杂化拖慢进度

控制策略：

- 第一批先固定“最小字段集”
- 入库/出库确认统一走 Service 层事务
- 权限收尾阶段只接最小 RBAC，不一次做满
- 每完成一个模块就更新接口文档并联调

## 12. 下一阶段预告

第一批完成后再进第二批：

- 库存调整
- 库存盘点
- 调拨管理
- 操作日志
- 基础报表

一句话标准：

第一批先把仓储主链路做稳，再把登录权限压轴接上，确保演示效果和后续扩展都不掉链子。
